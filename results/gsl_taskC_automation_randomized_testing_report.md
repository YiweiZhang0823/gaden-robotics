# 任务 C 汇报稿：完善自动化测试流程并引入随机化初始条件

## 汇报目的

本部分对应导师提出的第三点：在现有实验基础上，进一步完善测试流程，使实验可以更自动化地运行和记录结果；同时，不再把每一类初始条件只写成一个固定点，而是允许某些初始条件带有受控随机性。

这一部分的目标不是立刻扩展更多实验，而是把实验方法从：

> 测几个手工设定坐标点

推进到：

> 自动测试一类具有明确物理约束的初始条件

## 为什么需要任务 C

当前实验虽然已经比较系统，但仍有两个局限：

1. 某些初始条件是单个固定点，容易让人质疑结论是否只适用于这个坐标；
2. 虽然已经有命令生成脚本、批量记录和汇总 CSV，但自动化流程还主要是“人工触发多个固定实验”，还没有完全发展为“自动采样一组满足条件的初始状态并批量运行”。

导师提出加入随机性，是希望把“一个点”提升为“一类条件”。例如：

- 不是固定写成“距离气源 0.8 m”
- 而是写成“距离气源约为 `x`，允许在 `x-sigma` 到 `x+sigma` 范围内随机扰动”

这样实验研究的对象就不再是单个坐标，而是一类状态分布。

## 当前已经具备的自动化基础

目前系统中其实已经完成了第一层自动化：

### 1. 初始状态已经结构化保存

`initial_states.yaml` 已经把三种风场和五类初始状态统一整理成结构化配置：

- 场景 key
- 初始状态 key
- `base_pose`
- `ee_target`
- `source_relation`
- `plume_relation`
- `wind_relation`

这意味着实验设计已经不是完全散落在手工命令里，而是有统一数据源。

### 2. 已有自动命令生成脚本

目前已经有：

- `gsl_sensitivity_commands.py`
- `gsl_repeatability_commands.py`

它们能自动输出：

- 启动场景的命令
- 运行参数组的命令
- 输出 CSV 的路径
- 对应 log 文件路径

### 3. 已有自动记录与汇总

目前实验结果已经可以自动写入：

- `gsl_sensitivity_trials.csv`
- `gsl_repeatability_trials.csv`

并通过 summary 脚本自动生成汇总表。

### 4. 已能完成分阶段批处理

目前已经能自动化完成：

- 初始状态矩阵实验
- focused sensitivity experiment
- repeatability experiment

因此，任务 C 不是从零开始做自动化，而是在现有自动化之上再往前推进一步。

## 当前自动化流程还缺什么

尽管已经有自动化基础，但目前还缺少下面这一步：

> 从“固定初始点的自动运行”升级为“带物理约束的随机初始条件自动运行”。

也就是说，现在的自动化更像：

- 预先定义一个固定坐标
- 自动打印命令
- 运行并记录结果

而导师希望的下一步更像：

- 先定义一类状态条件
- 在这类条件内随机采样若干初始点
- 自动运行这些样本
- 自动记录每个样本的具体坐标与结果

## 什么叫“随机化初始条件”

这里的随机性不是无约束地乱取点，而是：

### 先定义物理条件，再在小范围内随机扰动

例如：

#### `far_near_edge`

原本是一个固定点：

```text
exp2_time_varying / far_near_edge = (1.45, 4.25, 0.65)
```

后续可以改成：

- 距离气源大致维持在当前量级
- 仍然要求接近羽流边缘
- 在一个小范围内随机采样

比如：

```text
x ~ [x0-sigma_x, x0+sigma_x]
y ~ [y0-sigma_y, y0+sigma_y]
z ~ [z0-sigma_z, z0+sigma_z]
```

但前提是采样后仍满足“edge”这一物理语义。

### 所以随机性的重点不是“随机”，而是“受控”

它必须满足两点：

1. 采样范围不能太大，否则状态语义会变掉
2. 每个随机样本都要能回溯，必须记录实际采样出来的坐标

## 任务 C 推荐的自动化结构

### 第一步：把固定状态升级为“状态模板”

当前 `initial_states.yaml` 中每个状态只有一个 `ee_target`。

下一步建议加一个“模板层”，例如：

- `nominal_ee_target`
- `sampling_sigma`
- `sampling_mode`
- `sampling_seed`

例如概念上可以写成：

| 字段 | 含义 |
| --- | --- |
| `nominal_ee_target` | 当前人工定义的中心点 |
| `sigma_xyz` | 允许扰动范围 |
| `sampling_mode` | 盒状采样 / 球壳采样 / 仅某方向扰动 |
| `semantic_guard` | 保证仍属于 far/off/edge/inside/upwind 这类状态 |

### 第二步：增加随机采样脚本

建议新增一个脚本，例如：

```text
gsl_sample_initial_states.py
```

它的作用是：

1. 读取场景与状态模板
2. 根据 `seed` 生成若干随机样本
3. 输出每个样本的实际 `ee_target`
4. 同时给每个样本一个 `sample_id`

### 第三步：增加批量运行脚本

建议再增加一个实验运行器，例如：

```text
gsl_batch_runner.py
```

它负责：

1. 按样本列表逐个运行实验
2. 自动写入结果 CSV
3. 把 `sample_id`、真实采样坐标、seed 一起记录

### 第四步：记录“模板”和“样本”两个层次

后续结果表中除了现在已有字段，还建议增加：

| 字段 | 含义 |
| --- | --- |
| `sample_id` | 第几个随机样本 |
| `sampling_seed` | 本次采样随机种子 |
| `nominal_ee_x/y/z` | 状态模板中心点 |
| `sampled_ee_x/y/z` | 实际随机采样坐标 |
| `sigma_x/y/z` | 本次允许扰动范围 |

这样后续分析时就可以区分：

- 这是哪一类物理状态
- 这个样本在该状态内偏了多少
- 偏移是否会影响成功率和路径长度

## 为什么这样做对论文更有帮助

如果只用固定点，别人可能会问：

> 你这个结论是不是只对 `(1.45, 4.25, 0.65)` 这个点成立？

如果用带约束的随机样本，你就可以回答：

> 不是只对一个点，而是对“远离气源且接近羽流边缘”这类状态，在小范围扰动下仍然成立。

这会让你的结论从：

```text
单点实验结果
```

提升为：

```text
一类初始条件的统计结果
```

这正是导师希望看到的“自动化 + 随机性”价值。

## 任务 C 建议先从哪几类状态开始

不建议一开始对全部 15 个状态都做随机化。

更合理的顺序是先挑最关键的两类：

### 优先级 1：`far_near_edge`

原因：

- 它在 `exp2` 和 `exp3` 中都很关键
- 是最容易体现参数敏感性和风场差异的边缘状态
- 小范围扰动后，最适合看成功率是否稳定

### 优先级 2：`far_inside_plume`

原因：

- 在 `exp3` 中，虽然能很接近气源，但确认不稳定
- 可以用随机扰动检查“inside plume”状态内部是否也存在明显随机性

因此，任务 C 的起步建议不是“大而全”，而是：

1. 先把 `far_near_edge` 随机化
2. 再考虑 `far_inside_plume`

## 任务 C 的阶段性目标

如果把任务 C 分成两小步，更容易执行：

### C1：流程自动化

先做到：

- 自动生成实验命令
- 自动记录样本坐标
- 自动输出统一 CSV

这一层先不一定非要加入随机采样，只要把批处理更顺即可。

### C2：受控随机化

再做到：

- 对一个状态模板采样多个样本
- 固定 seed 可复现
- 自动记录每个样本的实际位置

这样不会一下子把工程复杂度拉太高。

## 任务 C 的核心结论

本部分可以归纳成一句更适合汇报的表述：

> 当前实验已经具备命令生成、批量运行和结果记录的基础自动化能力。下一步计划不是盲目扩展更多固定点，而是将部分初始状态升级为带物理约束的随机状态模板，在保持“远/近气源、羽流外/边缘/内部、上游/下游”等语义不变的前提下，对初始坐标进行小范围随机采样，并自动记录样本坐标、随机种子和实验结果。这样可以把实验对象从单个点扩展为一类条件，从而提高结论的统计意义和可重复性。
